# Architecture — key modules & data flow

Moved from the repo CLAUDE.md (budget): the per-module reference for `app/`.
The one-line summary stays there; everything below auto-loads when editing
`app/**`, `tests/**`, or `alembic/**`.

### Key modules
- `app/main.py` — FastAPI app + lifespan. Logging is configured by **`app/logging_config.py:setup_logging()`** (#89): timestamped text, or single-line JSON when `settings.log_json` (`ICEBERG_EBS_LOG_JSON`) is set — replaces the old bare `basicConfig` (no timestamp). Unauthenticated ops probes (#26): **`/healthz`** (liveness — process up, no dependency checks) and **`/readyz`** (readiness — runs `SELECT 1`, returns 503 if the DB is unreachable; also reports `last_scheduler_run` from the in-process `app/scheduler_state.py` signal (no history scan on the probe; scheduler-only so an API fetch can't mask a stall) so an external monitor can catch a stalled scheduler — advisory, never flips the 200/503 — #89). Helm/Compose probes point at these. The dashboard (`routes/ui.py`) surfaces per-extension last fetch status/error and a fleet **"Fetch health"** count of watchlist extensions that are failing (latest `FetchLog.success` is false **and not** `store_outage`) or stale (no successful refresh within `max(2× fetch_interval, 60m)`) — a store-outage skip (#108) is excluded from "failing" so a store being down doesn't blame its extensions (a prolonged outage still surfaces via staleness)
- `app/config.py` — pydantic-settings `BaseSettings`, env prefix `ICEBERG_EBS_`
- `app/version.py` — `get_version()` (lru_cached): resolves the running build as `build N · sha` (`N` = `git rev-list --count --first-parent HEAD`, +1 per merge to main) in priority order `ICEBERG_EBS_VERSION` env → stamped `app/_version` file → runtime git → `"dev"`. Shown at the bottom of the rail and set as the FastAPI app `version`. See the Versioning section below
- `app/database.py` — async SQLAlchemy engine (PostgreSQL) with a tuned connection pool (`pool_size=5`, `max_overflow=10`, `pool_pre_ping`, `pool_recycle=1800`). **Schema is managed by Alembic** (D1): `init_db()` opens a connection and `run_sync`s `_run_migrations`, which upgrades an empty database to head from the baseline, or — for a database created the old way (tables present, no `alembic_version`) — **stamps** it at the *baseline* revision (without recreating anything) and then upgrades to head, so post-baseline migrations still run (#143; stamping head would have marked them applied without running them). A database already **falsely stamped** at head by the retired path (baseline schema behind a post-baseline stamp, detected via the still-naive `user.created_at` — the first post-baseline migration made it timestamptz) is re-stamped at the baseline and repaired the same way. A **stamped-but-empty** database (`alembic_version` present but the core `user` table missing — the state the test suite's `create_all`/`drop_all` used to leave a dev DB in, since `alembic_version` isn't a SQLModel table; #113) has its lying stamp reset to base so the upgrade rebuilds from scratch instead of trusting it and running nothing — inert in production, which never drops tables. Conversely an **unstamped database with tables present** (no `alembic_version`) is classified by matching its live schema against the models: if it **exactly matches head** (`_schema_matches_head` via `compare_metadata`) — what an *aborted* test run from the current checkout leaves behind, since conftest drops `alembic_version` in setup and `create_all` builds the head schema; #199 — it is stamped at **head** so the upgrade is a no-op, instead of being misclassified as pre-Alembic, stamped at the baseline, and re-running post-baseline migrations against columns `create_all` already made (`DuplicateColumn`, crashing `make dev`). An unstamped schema that is **post-baseline but does *not* match head** (`_has_post_baseline_schema` true via the still-naive-`created_at` discriminator, but `compare_metadata` non-empty — an aborted run from an *older* checkout, then the code updated) is **refused with a clear error** rather than stamped at head and silently skipping the intervening migrations (#199 review). Both are inert in production, which never runs `create_all` and whose genuine pre-Alembic DBs have the naive-`created_at` baseline schema. `tests/conftest.py`'s session fixture also drops `alembic_version` in setup + teardown so a `make test` no longer leaves the dev DB unbootable. The version INSERT is committed explicitly (`await conn.commit()`) because the connection isn't in autocommit, so the DML row would otherwise roll back when the connection closes. Migration env in `alembic/` (`env.py` reads `settings.database_url`, registers `SQLModel.metadata`, and reuses the startup connection via `config.attributes["connection"]`); baseline revision in `alembic/versions/`. **Adopting Alembic assumes existing production DBs are already on the final pre-Alembic schema** (they are — the retired `_migrate_*` ran on every prior startup), which corresponds to the baseline revision, before being stamped. Add new schema changes with `alembic revision --autogenerate`; `tests/test_migrations.py::test_head_matches_models` fails if the baseline/head drifts from `models.py`.
- `app/models.py` — SQLModel table definitions (`User`, `Extension`, `FetchLog`, `InstallCountHistory`, `InstallObservation`, `AlertDestination`, `AlertRule`, `AlertLog`, `ApiKey`). Timestamp columns use `sa_column=_tz_column(...)` (`timestamptz`) because the app writes tz-aware UTC datetimes. **Referential cleanup lives in the schema** via FK `ondelete` (the SQLModel-recommended pattern): history-preserving FKs are `SET NULL` (`AlertLog.{rule_id,destination_id,user_id}`, `Extension.user_id`), child rows removed with their parent are `CASCADE` (config rows + per-extension `FetchLog`/`InstallCountHistory`/`InstallObservation`/`AlertRule`/`AlertLog`). The delete handlers rely on these instead of hand-severing FKs — see the FK migration `…_fk_ondelete.py`. **`InstallObservation`** (#29) is SOAR-fed org inventory: one row per `(extension_id, asset_id)` (unique constraint → re-pushes upsert `last_seen`), carrying `asset_type`/`department`/`source`/`first_seen`. `Extension.install_footprint` caches the distinct-asset count, maintained by `POST /api/inventory`; **exposure** ("blast radius") = `risk_score × install_footprint` is always **derived** (never stored). **`FetchLog.store_outage`** (#108) marks a failure row written when the per-store circuit breaker skipped the extension (store down), so the Fetch-health tile can tell that apart from a broken extension. **`Extension` JSON-in-str columns** (`permissions`, `risk_detail`, `package_analysis`, `pending_alert_events`) are read through a single defensive parse each (missing / unparsable / wrong-shape → safe fallback) via `utils.json_list`/`json_object`, so consumers never re-hand-roll it and can't reintroduce the #17/#61 bug class (#167). Three are typed accessor methods on `Extension` — `permissions_list()`, `analysis_dict()`, `risk_detail_dict()`. The fourth, `pending_alert_events`, is decoded by **`services._parse_pending_events(raw)`** (not a model method — it returns typed `ChangeEvent`s from `notifications.py`, which the model can't import without a cycle): it drops both non-dict entries **and** malformed event dicts, so a corrupt/partial marker can't crash the fetch or re-fire forever (#197). Writers still `json.dumps` back onto the column
- `app/auth.py` — itsdangerous signed cookies, `require_auth` / `require_admin` FastAPI dependencies; `verify_credentials` always runs bcrypt (via `_DUMMY_HASH`) even for unknown usernames to prevent timing-based user enumeration. `hash_password` / `verify_password` are **async** — bcrypt is ~100ms of pure CPU, so they offload to a worker thread via `anyio.to_thread.run_sync` (single uvicorn worker: running it inline stalls every request + the scheduler). All callers must `await` them; the sync cores are `_hash_password_sync` / `_verify_password_sync`. `require_api_auth` **throttles** the `ApiKey.last_used_at` write (only when missing or older than `settings.api_key_last_used_throttle_seconds`, default 60s) — a commit per request (including read-only GETs) is a wasted write that contends with the scheduler under load. **Session revocation (M1):** every cookie carries a signed issued-at (read via `get_session_claims`); `require_auth`/`require_api_auth` reject cookies older than the user's `User.password_changed_at` (`_session_after_password_change`, 1s tolerance for timestamp granularity), and `change_password` bumps that marker **and** deletes the user's API keys — so a password reset invalidates other-device sessions and leaked bearer tokens. **`require_admin_ui`** is the HTML-admin counterpart to `require_admin`: built on `require_auth` it gives 303 redirects (to `/login` if unauthenticated, `/` if non-admin) instead of raw JSON 401/403
- `app/fetchers/` — one fetcher class per store; `get_fetcher(store, client)` factory in `__init__.py`. Package download is best-effort: `fetch()` (base + vscode/edge overrides) catches **only** `(FetchError, httpx.HTTPError)` → logs + `package=None`; genuine programming errors propagate instead of being swallowed into a silent midpoint-fallback score (M5). **`transport.py` (#108)** holds `RetryTransport`, the shared client's resilience layer: it retries **transient** failures (connect/timeout + `RETRYABLE_STATUS` = 429/500/502/503/504) on **idempotent** methods only, with exponential backoff + full jitter, honouring `Retry-After` (bounded by `_MAX_SLEEP_SECONDS` so a hostile value can't wedge the sequential scheduler). 404 is never retried (delisted → surface now); POST is never retried (webhook delivery must not be silently re-sent, and its call site already disables redirects). `transport.py` also holds **`ProxyRoutingTransport`** (#216): routes each request direct or through the configured outbound proxy by consulting `app/proxy.py`'s in-memory snapshot **per request** (so an admin edit applies on the next request — no client rebuild), lazily building the proxied inner transport keyed on the credential-injected proxy URL and **retiring (not closing)** the old one on change (in-flight requests finish; everything closes in `aclose()`). It's a transport (not httpx `mounts=`) because NO_PROXY CIDR entries can't be URL patterns. Deliberately no logging inside it — the resolved proxy URL carries credentials. `app/main.py` builds the chain `RetryTransport(ProxyRoutingTransport(limits=…))` — retry OUTER so each attempt re-routes; limits go on the **innermost** transports because httpx ignores `AsyncClient(limits=…)` when a custom `transport=` is supplied (direct + proxied pools each carry `limits`, so the theoretical cap is 2×)
- `app/inspector.py` — static analysis of downloaded zip packages (CRX/VSIX). `inspect_package()` is pure-CPU (~20 regexes over up to 500 JS files); `services.py` calls it via `anyio.to_thread.run_sync` so it doesn't stall the event loop / scheduler. `PackageAnalysis.to_json_dict()` (what `services.py` persists to `Extension.package_analysis`) and `PackageAnalysis.stored_defaults()` (what `routes/ui.py` backfills onto a sparse/old stored blob before rendering) are the **single home** for the stored-field list — both derive from the dataclass fields (excluding the internal `_finding_keys` and the transient `version`/`author` manifest fallbacks), so a new analysis field is one dataclass edit and can't drift between store and render (#164); `tests/test_inspector.py` asserts the two key-sets stay identical
- `app/permissions.py` — single source of truth for the permission-tier sets (`CRITICAL_PERMISSIONS`, `HIGH_PERMISSIONS`, `MEDIUM_PERMISSIONS`, `BROAD_HOST_PATTERNS`). Imported by both `scoring.py` (score) and `inspector.py` (findings) so the two can't drift (#63) — add a permission to a tier here, never re-inline these sets
- `app/scoring.py` — pure scoring functions, `compute_risk_score()` → `RiskDetail` NamedTuple; `risk_level(score)` is the single source of truth for the score→band thresholds (75/50/25), reused by `routes/api.py` and `notifications.py` — do not re-inline those thresholds elsewhere. Permission tiers come from `app/permissions.py`
- `app/services.py` — `fetch_and_store(ext, session, client)`: shared pipeline used by both API routes and the scheduler; stages changes but does **not** commit and does **not** fire alerts — it returns `(ext, events)`. `fire_pending_alerts(events, ext, engine, client)`: the caller invokes this **only after committing**. Firing is deferred on purpose: `fire_alerts` opens its own second DB session, which must not run inside the caller's still-open write transaction (it would contend with / be isolated from the caller's uncommitted writes). Never call `fire_alerts`/`fire_pending_alerts` while the caller's write transaction is open. After commit, callers `session.refresh(ext)` (reloading expired attrs) before firing, since `fire_alerts` reads `ext` to build the payload. **Recoverable alerts (#109):** `fetch_and_store` stages the detected events on `Extension.pending_alert_events` (JSON) in the **same commit** as the state change, **merging** with any still-undelivered events already in the marker rather than overwriting them (a prior failed delivery leaves them for retry, and the manual-refresh path doesn't run recovery first). It returns the full merged set so the caller fires everything pending. `fire_pending_alerts` clears the marker only after `fire_alerts` returns (delivered + logged), and only via **compare-and-clear** (`_clear_pending_alerts(ext_id, engine, expected)` clears iff the stored marker still equals `expected`) so a concurrent refresh's appended events aren't wiped. `expected` is the raw marker string the events were derived from: the normal path re-serialises the (unfiltered) fired events — an identical string — while recovery passes the **raw** marker explicitly (`fire_pending_alerts(..., expected_marker=raw)`), because it may have delivered a *filtered subset* of a corrupt marker and must still clear the whole thing, or it never matches and re-fires every cycle forever (#197). If the process dies between commit and delivery, the marker survives and **`recover_pending_alerts(engine, client)`** — run at the head of each `refresh_watchlist` cycle, deliberately **not** in the lifespan at startup (running it inline there would block uvicorn from binding / answering probes for up to one webhook timeout per pending extension, risking a liveness-probe kill mid-recovery; deferring to the scheduler keeps startup fast **and** keeps recovery running in exactly one place so it can't race a concurrent refresh's delivery of the same events — #155; the durable marker makes the ≤ `fetch_interval_minutes` deferral safe, nothing is lost) — decodes it via `_parse_pending_events` (dropping any junk) and re-fires it (idempotent). **Graceful drain:** APScheduler 3.x's `AsyncIOExecutor.shutdown(wait=True)` *cancels* running async jobs rather than awaiting them, so the lifespan instead `scheduler.pause()`s, then `await drain_inflight(settings.shutdown_drain_seconds)` (which awaits the in-flight `refresh_watchlist` task tracked in `scheduler._inflight`), then `scheduler.shutdown(wait=False)`. The durable marker is the backstop past the container grace period (`terminationGracePeriodSeconds`/`stop_grace_period`, kept above `shutdown_drain_seconds`).
- `app/notifications.py` — `detect_changes()` + `fire_alerts()`: compares old/new extension state and delivers to matching destinations by **dispatching through the sender registry** (`app.senders.get_sender(dest.kind).send(...)`, #37 — not `send_webhook` directly), so a rule can fire to a webhook/Slack/Teams/email/Jira/ServiceNow destination unchanged; `fire_alerts()` takes an `AsyncEngine` and commits `AlertLog` rows in its own dedicated session, independent of the caller's transaction. It must run **after** the caller has committed (see `fire_pending_alerts`) so its session does not run inside the caller's open write transaction. It groups matching rules by event type and delivers **every** event against its rules — including multiple events of the same type, which the merged pending marker can hold (e.g. `new_version` 1.0→1.1 then 1.1→1.2); it must **not** collapse to one event per type or the older undelivered event is dropped and then cleared with the marker (#144). The `permission_change` event diffs **both** API permissions (`ext.permissions`) **and** host permissions (read from the stored `package_analysis` via `_host_permissions`) — gaining broad host access like `<all_urls>` must alert (#60); both fields are only rewritten on a fresh successful inspection, so a transient download failure can't fire a spurious change. Per-delivery it builds an `AlertMessage` (via `senders.webhook.extension_deep_link` for the app link) and lets the kind's sender render the wire shape; the delivery error is `proxy.scrub`'d + truncated before it reaches an `AlertLog.error` (#228). **`build_alert_payload(...)`** — the canonical webhook payload shape (`{text, event, extension:{id,name,store,store_url,[iceberg_ebs_url]}, change:{old,new}, risk_score}`) — now lives in **`app/senders/webhook.py`** (re-exported here for back-compat) so `senders` needs no import back into `notifications`; the webhook **and** test paths both render through the sender, keeping "test == real" by construction (#168)
- `app/senders/` — **outbound alert-delivery adapters (#37)**, the delivery-side mirror of `app/oidc/`. `base.py` (pure, mypy-clean) defines the `AlertSender` Protocol, the `AlertMessage`/`ConfigField` value types, the `SenderError`/`DestinationConfigError` exceptions, and the self-registering registry (`register_sender`/`get_sender`/`all_senders`/`sender_kinds`/`kind_descriptors`; display order fixed by `_KIND_ORDER`, independent of import order). `http.py` holds **`HttpJsonSender`** — the shared "POST a rendered JSON body through `webhooks.send_pinned_request`" core (the `StandardOIDCAdapter` analogue): a concrete kind is just a `render_payload(message, config)` renderer plus its `config_fields`, and inherits SSRF-validate-at-create+send, IP pinning, disabled redirects, proxy routing and no-POST-retry for free. Six peer kinds self-register (webhook/slack/teams/jira/servicenow are HTTP; **webhook is not privileged** — Slack/Teams are specialised webhooks, Jira/ServiceNow add an `Authorization` header + an appended API path): `webhook.py` (canonical payload), `slack.py` (Block Kit), `teams.py` (Adaptive Card), `tickets.py` (Jira ADF create-issue + ServiceNow table create). `email.py` is the one non-HTTP transport — SMTP via stdlib `smtplib` offloaded through `anyio.to_thread` (it deliberately does **not** traverse the outbound proxy); unavailable (creation refused) unless `ICEBERG_EBS_SMTP_*` is configured. **Secrets are env-only** like the OIDC/proxy credentials: SMTP creds are `settings.smtp_*`; per-destination Jira/ServiceNow API tokens are referenced by an uppercase **`secret_ref`** (stored on the row, not secret) and read at send time from `ICEBERG_EBS_DEST_SECRET_<REF>` — validated to resolve at create/update time. The kind set is single-sourced from the registry for the API/UI; the `ck_alertdestination_kind` CHECK on `AlertDestination` is its schema backstop, held in lockstep by `tests/test_senders.py::test_model_check_constraint_matches_registry`
- `app/webhooks.py` — `validate_webhook_url()` (SSRF validation: scheme/blocklist/IP-range checks + **rejects embedded userinfo** `user:pass@host` — the pinning rebuild would drop it silently and `AlertDestination.target` is persisted/API-returned, so a URL credential would be stored plaintext + exposed, #79; credentials go in a sender `secret_ref`, never the URL — returns the resolved public IPs) and **`send_pinned_request()`** (the single shared delivery core: validates + resolves + connects to the pinned IP, preserving the original `Host` header and TLS SNI, merging caller headers with the pinned `Host` always winning — every HTTP sender kind delivers through this). DNS resolution is isolated in `_resolve_host()` so tests can stub it. `app/senders/http.py` (and thus all HTTP delivery) sends through this module
- `app/scheduler.py` — APScheduler `AsyncIOScheduler` background watchlist refresh; each extension is processed in its own `AsyncSession` + commit so a single failure cannot corrupt or roll back changes for the entire batch. `_refresh_one` returns an `_Outcome` (SUCCESS/FAILED/GONE) that drives a per-cycle, per-store **`_StoreCircuitBreaker`** (#108): a store's counter resets on any success, so an isolated broken extension (404) never trips it, but `settings.store_circuit_failure_threshold` **consecutive** failures open the store's circuit and the cycle's remaining same-store extensions are skipped and logged as `FetchLog.store_outage=True` (via `_record_store_outage`) rather than each looking broken. Also registers a **daily retention prune** (`run_retention_prune`) — only when `settings.retention_days > 0`, firing once at startup (`next_run_time=now`, #145). Both jobs set `misfire_grace_time=None` so a busy/stalled single-worker loop can't silently drop a due fire under APScheduler's 1s default — in particular the prune's at-startup fire on a CPU-starved restart (#198)
- `app/retention.py` — data-retention pruning (#22) + install-footprint decay (#287). `prune_expired(session, retention_days, now=...)` deletes `FetchLog`/`InstallCountHistory`/`AlertLog` rows older than the window and `InstallObservation` rows whose `last_seen` is (cutoff = `now − retention_days`), returns per-table delete counts, and is a no-op when `retention_days <= 0`; `Extension` rows are never touched. `run_retention_prune()` is the scheduler entry point — own session + commit, gated by `ICEBERG_EBS_RETENTION_DAYS` (default 0 = disabled). **Footprint decay (#287):** `freshness_cutoff()` is the single home of the `ICEBERG_EBS_INVENTORY_FRESHNESS_DAYS` window (default 30; 0 disables); observations not re-seen within it stop counting toward `install_footprint`. Applied at the inventory API's per-batch recompute, the detail-page department breakdown, and `refresh_install_footprints()` — a bulk correlated UPDATE over every extension with observations or a set footprint, run by the daily `run_footprint_refresh` scheduler job (ungated by retention; it covers extensions that stopped appearing in SOAR pushes entirely, which the per-batch recompute never touches)
- `app/proxy.py` / `app/proxy_settings.py` / `app/routes/proxy.py` — **outbound proxy (#216)**. `proxy.py` is the pure resolver (typed, mypy-clean): `resolve_proxy_url(cfg, url) -> str | None` (None = direct) over three modes — `SYSTEM` (default; parses `HTTP(S)_PROXY`/`ALL_PROXY`/`NO_PROXY` from the env itself, because httpx never applies env proxies through a custom `transport=`), `NONE`, `EXPLICIT` (configured URL unless the target matches the no-proxy list — standard NO_PROXY semantics incl. CIDRs, `_should_bypass`). Routing is read from an in-memory `ProxyConfig` snapshot (`get_config`/`set_config`; `None` = unloaded → direct, pre-feature behaviour) consulted per request by `ProxyRoutingTransport` (see `app/fetchers/` above), so **all** egress — store fetchers, package downloads, webhook delivery — inherits it. `proxy_settings.py` manages the admin-editable `ProxySettings` DB singleton (id=1), lazily seeded from the `ICEBERG_EBS_PROXY_*` env on first read; every save pushes a fresh snapshot. `routes/proxy.py` is the admin JSON API (GET/PUT settings — PUT is `extra="forbid"` so credential keys 422, rejects a userinfo `proxy_url` (creds would persist onto the API-visible row), and enforces the EXPLICIT⇒URL invariant against the **resulting** settings, not just the request fields — clearing only `proxy_url` while the stored mode is explicit would otherwise silently fail open to direct; GET `/api/proxy/targets` returns **labels only**; POST `/api/proxy/test` dials a server-known target by label — never a body URL, which would be an SSRF oracle — with `follow_redirects=False` always, the full validate+pin delivery treatment for user-controlled webhook origins, and returns `ok: HTTP n` or `error: <ExceptionClassName>` only). **Credentials are env-only** (`proxy_username`/`proxy_password`): injected into the proxy URL at resolution time (`_with_credentials`), never on the DB row, never in API responses, scrubbed from log lines (`proxy.scrub`); `validate_proxy_settings` also rejects a userinfo env URL. Webhook interplay: `send_webhook` pins the URL netloc to the validated IP, so the routing decision (and any no-proxy bypass) runs against the **IP** — bypassing a webhook destination needs an IP/CIDR entry, and the pinned CONNECT preserves the DNS-rebinding defence through the proxy. `validate_proxy_settings` runs first thing in the lifespan (fail-fast; error text never echoes the URL), and the startup `refresh_cache` is **fail-closed**: a load failure aborts startup rather than leaving the snapshot unloaded (unloaded = all-direct, which would let an EXPLICIT deployment silently bypass its mandated proxy). Admin page at `/admin/proxy` (changes apply on the next outbound request — single worker, in-process snapshot)
- `app/oidc/` / `app/oidc_settings.py` / `app/routes/oidc.py` / `app/routes/oidc_settings.py` — **SSO via OIDC (#32)**, ported from the sibling deep_thought project (its #25) to stay idiomatic with the app family. `app/oidc/base.py` (pure, mypy-clean) defines `OIDCIdentity` + the `OIDCAdapter` protocol/registry; adapters self-register at import — `entra.py` is the only custom one (requires `tid` → immutable `auth_tenant`, `email`→`preferred_username` fallback, `xms_edov` email-verified policy), authentik/auth0/okta share `StandardOIDCAdapter`. `app/oidc/config.py` (pure, mypy-clean) holds `OIDCProviderConfig`/`OIDCRuntimeConfig` (+ per-provider discovery-URL construction), `EDITABLE_FIELDS`, and `validate_config()` — auth-mode/URL/role-map validation, `openid`-scope + env-secret checks per enabled provider, the Entra multi-tenant-alias rejection, and the **lockout guard** (`auth_mode=oidc` with zero complete providers is refused). `app/oidc/service.py` wires Authlib: process-wide `OAuth()` registry (PKCE forced via `code_challenge_method=S256`; discovery via `server_metadata_url`), `ensure_registered()` for lazy registration, and `reset_registration()` which **rebinds a fresh `OAuth()`** on every settings change — Authlib's `create_client` serves a cached client, so re-registering alone would keep stale config. OIDC egress (discovery/JWKS/token/userinfo) goes through its own `RetryTransport(ProxyRoutingTransport)` chain passed via `client_kwargs["transport"]`, wrapped in `_NonClosingTransport` because Authlib `aclose()`s a throwaway client per call (closing the shared chain after the first callback); the real chain closes in lifespan shutdown. `provision_oidc_user()` is the match/JIT policy: match on the validated **`(oidc_issuer, oidc_subject)`** **scoped to the configured provider** (`auth_provider`, row-locked). The issuer stays in the key so a re-pointed adapter can't inherit an account (#218 review), and the provider scope stops a hostile/compromised configured provider from spoofing another's issuer (Authlib only validates `iss` against that provider's *own* discovery metadata) and taking over its accounts — the same `(issuer, subject)` presented under a **different** provider is refused as an `"identity conflict"`, backed by the global `uq_user_issuer_subject` constraint (#226). Then require a **verified** email and a DB-level **partial unique index on SSO email** (`WHERE oidc_subject IS NOT NULL`) so two concurrent first-logins can't duplicate an account, deny **any** email collision ("account linking required" — mutable email never links, so an IdP email claim can't claim a local admin), else JIT-create with `password_hash=None`, `role_managed_by_idp=True`, username = normalized email (residual username collision → deterministic sha-suffix). `map_is_admin` is **any-match** over the `group=admin|user` allowlist (values widen to the role enum with #33); sync on returning login is gated on `role_managed_by_idp` (the seeded break-glass admin keeps `False`, so the IdP can never demote it) and bumps `password_changed_at` (the generic session cutoff) on change. **Entra groups-overage** (>~200 groups → the ID token omits inline group/role membership and emits the distributed-claims pointers `_claim_names`/`_claim_sources` instead — the overage is **always keyed on `groups`** in `_claim_names`, even under `emit_as_roles`, so a configured `role_claim` of either `groups` or `roles` is affected) is detected in `EntraAdapter` (`_role_claim_overaged` in `base.py`, gated on a configured `role_claim` that was **not** delivered inline) and **fails the login closed** (a logged `ValueError` → `/login?error=sso`) rather than reading the absent claim as "no groups" and silently demoting the admin + revoking their sessions (#227); an inline value is trusted over the pointer (no false-positive deny), and `_groups_from` stays a non-raising normaliser. `app/oidc_settings.py` mirrors `proxy_settings.py`: env-seeded `OIDCSettings` singleton (id=1), in-memory snapshot, `update_settings` validates the **resulting** config under `with_for_update, populate_existing` (TOCTOU discipline, #217 lesson), startup `refresh_cache` is **fail-closed**. Routes: `/auth/oidc/{provider}/login|callback` mint the same `set_session` cookie as password login (all downstream authz unchanged); failures 303 to `/login?error=sso` and never log the code/token. Handshake state (state/nonce/PKCE verifier) lives in a dedicated 10-min `SessionMiddleware` cookie `iceberg_ebs_oidc` (added innermost in `main.py`). `/auth/oidc/*` joins the login rate-limit zone. Admin page `/admin/oidc`; client secrets are env-only (`ICEBERG_EBS_OIDC_<P>_CLIENT_SECRET`). SAML, account linking, and RP-initiated logout are deliberate follow-ups
- `app/ratelimit.py` — `LoginRateLimiter` (process-local `login_limiter` singleton) for app-level login throttling (M3), independent of the edge proxy: counts failures per (client IP + username), locks the pair out for `settings.login_lockout_seconds` after `settings.login_max_attempts` within `login_attempt_window_seconds`. In-process state is fine because the deployment mandates a single worker. `login_post` returns 429 + `Retry-After` while locked and `reset()`s on success. Also `RequestRateLimiter` (token-bucket) backs **app-side edge rate limiting** — the replacements for the old nginx `api` **and** `login` `limit_req` zones, added because stock Caddy has no `rate_limit` directive. The `edge_rate_limit` middleware in `app/main.py` applies two singletons keyed on `request.client.host` (the Caddy-set canonical `X-Forwarded-For`, spoof-proof per #77): `api_limiter` over `/api/*` (#188, gated by `settings.api_rate_limit_enabled`) and the tighter `login_request_limiter` over `POST /login` (#196, gated by `settings.login_rate_limit_enabled` — its **own** switch, so disabling API limiting can't silently drop login protection; the failure-keyed `LoginRateLimiter` above only locks an (IP, username) pair after N failures and so stops neither username-spraying nor a bcrypt-cost flood). Both default **off** so the test suite isn't throttled; the Compose/Helm prod env enables them, with the cluster ingress limiting at the true edge too
- `app/deps.py` — reusable FastAPI dependency aliases (`SessionDep`, `CurrentUser`, `WebUser`, `AdminUser`, `AdminUserUI`) used across route signatures instead of repeating `Annotated[..., Depends(...)]`. Each `/api` router declares its own `prefix="/api"` + `tags=[...]` on the `APIRouter` (so `main.py` just `include_router(...)` with no prefix, and `/docs` is grouped by tag). Endpoints declare their public schema as a **return-type annotation** (`-> ExtensionOut`, `-> list[UserOut]`, …) rather than the `response_model=` decorator arg — the FastAPI-preferred form, since the handlers already return the DTO. Also home to **`get_owned_or_404(session, model, obj_id, user_id, *, detail="Not found")`** (#169): the single owner-scoped fetch gate for the JSON API — loads the row and 404s if it's missing **or** owned by someone else (indistinguishable, so ownership can't be probed), replacing the ~12× hand-inlined `session.get(...) + if not obj or obj.user_id != …` across `routes/api.py`/`alerts.py`/`keys.py`. HTML routes that redirect-and-flash on a miss (`routes/ui.py`) keep their own handling
- `app/extension_queries.py` — the shared extension filter/sort/count query layer (#68, #163): the **`ExtensionFilters`** dataclass, **`build_extension_query(user_id, filters)`**, `count_rows(session, stmt)`, the `RISK_BANDS` (level → score range, same 75/50/25 thresholds as `scoring.risk_level`) and `SORT_COLUMNS` maps, plus `EXPOSURE_EXPR` / `exposure()` (exposure = `risk_score × install_footprint`, derived never stored, #29 — the SQL expression and its Python mirror live side by side). One public home consumed by the list API, the export endpoint and the dashboard so the three call sites can't drift — previously these lived as private names inside `routes/api.py` with `routes/ui.py` importing them (#163)
- `app/utils.py` — small shared helpers: `domain_from_url()`, and the defensive JSON-parse primitives for the JSON-in-str `Extension` columns — `safe_json_loads()` (#17 — logs + falls back instead of 500-ing on unparsable JSON) plus `json_list()` / `json_object()` (#167 — add a shape guard on top, so valid-but-wrong-shape JSON also falls back instead of `AttributeError`-ing at the consumer, #61/#150). These are consumed **through the `Extension` typed accessors** (`permissions_list()` / `analysis_dict()` / `risk_detail_dict()`) and **`services._parse_pending_events()`** (for `pending_alert_events`, #197), which are the single home for the parse — routes/services/notifications call those, not `json.loads` on the columns (#167)
- `app/routes/api.py` — JSON API routes for extensions (`/api/extensions/...`). `ExtensionOut.from_db(ext, include_threat_intel=...)`: the list endpoint passes `include_threat_intel=False` to skip the O(extensions × domains/URLs) VirusTotal/OTX indicator build it never renders; single-extension views keep the default `True` (D2). **`GET /api/extensions` is paginated + filterable (#23):** returns a `PaginatedExtensions` envelope `{items, total, limit, offset}` (not a bare list) and accepts `store`/`risk`/`publisher`/`q` (search over name/publisher/id, LIKE-escaped)/`sort`/`order`/`limit` (≤200)/`offset`. The filter+sort logic lives in **`app/extension_queries.py`** (see above); the list + export endpoints collect those params via the **`extension_filters` FastAPI dependency** (`FilterParams`) declared once (typed Query params → 422 on bad input); the dashboard builds the same dataclass from its own coerced strings (tolerating junk). **`GET /api/extensions/export?format=csv|json` (#25):** streams the full (filtered, unpaginated) set of `EXPORT_FIELDS` (score + key fields, no nested findings/threat-intel) with a `Content-Disposition` attachment; shares the same filter/sort params. **Route order matters** — `/extensions/export` and `/extensions/bulk` are registered before `/extensions/{ext_id}` so `"export"`/`"bulk"` aren't parsed as int ids. **`POST /extensions/bulk` (#24):** enrolls many extensions in one request (cap `MAX_BULK_ITEMS=100`), accepting structured `items` and/or a pasted `text` blob (`store,id` per line or store URLs via `_detect_store`/`_parse_bulk_text`). Each entry runs through the shared **`_enroll_extension`** primitive (validate → dedupe → create + `_fetch_and_score`, discarding the placeholder on a failed first fetch) — the same helper `add_extension` now uses — and returns a per-entry status (added/duplicate/invalid/error) + tallies. The loop captures `current_user.id` **before** processing because each enroll commits (expiring the ORM attribute). **`POST /api/inventory` (#29):** bulk-upserts SOAR install inventory (cap `MAX_INVENTORY_ITEMS=1000`); each observation resolves its extension through the same `_enroll_extension` primitive called with **`score=False`** (#78) — so an **unknown** extension is auto-enrolled onto the watchlist but its **scoring is deferred to the scheduler** (which scores every `watchlist=True` extension on its next run), keeping a large batch of unknown extensions from doing hundreds of sequential store fetches inside one request. Newly-created rows report status `deferred`; already-tracked ones report `observed`. `_upsert_observation` then writes/refreshes an `InstallObservation` keyed on `(extension, asset)` via a Postgres `INSERT … ON CONFLICT DO UPDATE` (#76). After the batch, each touched extension's cached `install_footprint` (distinct asset count) is recomputed in one pass. **Exposure** is sortable everywhere via `sort=exposure` — `SORT_COLUMNS["exposure"]` is the SQL expression `EXPOSURE_EXPR = Extension.risk_score * Extension.install_footprint` (NULL when either factor is, so existing nullslast handling applies; both in `app/extension_queries.py`); `ExtensionOut`/`EXPORT_FIELDS` expose `install_footprint` + `exposure` (computed by the `exposure()` helper mirroring the SQL)

### Data flow
1. Caller (API route or scheduler) calls `fetch_and_store(ext, session, client)`
2. `fetch_and_store` calls `fetcher.fetch(extension_id)` → `(ExtensionMetadata, bytes | None)`
3. If package bytes are present, `inspect_package()` runs static analysis
4. `compute_risk_score()` calculates the risk breakdown
5. Extension record, FetchLog, and InstallCountHistory are staged; `detect_changes()` compares the pre-fetch snapshot against the updated record; `fetch_and_store` returns `(ext, events)`
6. The caller **commits**, `session.refresh(ext)`, then calls `fire_pending_alerts(events, ext, engine, client)` → `fire_alerts()` POSTs webhooks and commits `AlertLog` rows in its own session, decoupled from the caller's transaction. This ordering is mandatory: `fire_alerts`' second session must not run inside the caller's open write transaction
