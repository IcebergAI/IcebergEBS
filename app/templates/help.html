Help

A quick reference for the risk model, ID formats, and webhook payloads.

What is IcebergEBS?

IcebergEBS tracks browser and editor extensions and scores them for risk. It downloads the actual extension package and inspects the code, combined with store metadata, to produce a 0–100 risk score across six signals.

Adding extensions

Paste either a full store URL or a bare extension ID on the add page. The store is auto-detected from URLs. To add many at once, use bulk import — paste one store,id (or store URL) per line. The dashboard list can be filtered by store and risk, sorted by any column, and exported to CSV or JSON. Search from the box in the top bar (press / to focus it) — it keeps the filters and sort you already have applied.

Risk score signals

Each extension is scored 0–100 across six signals. Higher = riskier.

Inventory & exposure

Risk alone doesn't tell you blast radius. Push your org's real install footprint from your SOAR (IcebergEBS doesn't build EDR/Intune connectors itself) to POST /api/inventory — a batch of {store, extension_id, asset_id, asset_type?, department?} observations. Each observation is upserted per (extension, asset), and any extension IcebergEBS isn't already tracking is auto-enrolled onto the watchlist and scored on the next scheduled refresh, so pushing inventory grows the watchlist automatically without stalling large batches on live store fetches.

Exposure ("blast radius") = risk score × assets installed. The dashboard ranks your highest-exposure extensions, and each extension's detail page shows how many assets it's on and a per-department breakdown. Sort any list/export by sort=exposure. Footprint decays: an observation your SOAR stops re-reporting drops out of the count after ICEBERG_EBS_INVENTORY_FRESHNESS_DAYS (default 30), so an extension removed from your endpoints stops inflating exposure.

Detection findings

IcebergEBS records structured findings while inspecting the downloaded package. Findings are deterministic static heuristics — they do not call external reputation services or require third-party API keys.

Threat Intelligence

IcebergEBS extracts observable indicators from the inspected package, including package hashes, external domains, and JavaScript network callouts. Referenced URLs that are only found as literal strings are kept separate from callouts. The Threat Intelligence panel provides copy controls and manual lookup links for external tools such as VirusTotal and AlienVault OTX.

Lookups are manual and privacy-preserving by default. IcebergEBS does not call external threat-intelligence APIs or submit packages; clicking an external lookup link sends the selected indicator to that service.

Users & accounts

IcebergEBS is multi-user. Each user maintains their own isolated list of monitored extensions — adding for one user has no effect on another.

  • Admin creates users via admin / users. There is no self-registration — except just-in-time provisioning on single sign-on (below).
  • Change your password or pick a theme (system, light, or dark) under the username menu in the top bar.
  • Admins can delete users — their alert rules, webhook destinations, and API keys are removed, while their extensions are kept (unassigned and taken off the watchlist) so fetch and alert history survives.

Single sign-on (OIDC)

IcebergEBS can sign users in against an OIDC identity provider — Microsoft Entra ID, Authentik, Auth0, or Okta — alongside or instead of password login. Admins configure providers live at Single sign-on in the rail; the ICEBERG_EBS_AUTH_MODE / ICEBERG_EBS_OIDC_* environment variables seed the initial values, and the per-provider client secrets are environment-only (never stored in the database or shown in the UI).

  • A first SSO login creates the account automatically (username = the IdP email). SSO accounts have no local password — password changes are the identity provider's job.
  • Admin membership can follow IdP groups: map group=admin pairs in the provider's role map. Unmapped users sign in as regular users, never admins, and the flag is re-checked on every SSO login (removing the group demotes and signs out other sessions).
  • The authentication mode is both (password + SSO), local (password only), or oidc (SSO only). OIDC-only mode is refused unless a complete provider is enabled, so a bad config can't lock everyone out; local accounts stay available as break-glass in both mode.
  • Accounts are keyed on the provider's immutable subject, never the (mutable) email — an SSO login whose email matches an existing account is denied rather than auto-linked.
  • SSO sessions are shorter-lived than password sessions and re-authenticate through the identity provider on expiry, so an IdP-side disable or password reset takes effect without an admin. Signing out also ends the session at the provider (RP-initiated logout).

API keys

Programmatic callers (SOAR pushing to POST /api/inventory, reporting scripts hitting the export endpoint, …) authenticate to the JSON API with a bearer token instead of a session cookie: Authorization: Bearer <key>. Create and revoke keys under account / API keys.

  • The full key is shown once at creation — only a prefix/suffix is stored for display.
  • Creating a key requires signing in interactively — a bearer token cannot mint another key, so a key can't renew itself past its lifetime.
  • Keys can be read-only: they work for GET requests and are rejected (403) for anything state-changing.
  • Changing your password revokes all of your API keys along with other-device sessions.
  • Keys owned by single sign-on accounts expire a bounded number of days after creation (default 30; ICEBERG_EBS_API_KEY_SSO_MAX_AGE_DAYS) — an IdP-side offboarding can't reach the app, so the lifetime bound is what contains those tokens. Re-create the key to keep an integration running.

Alerts & integrations

Configure destinations and alert rules under account / alerts. A rule fires when a monitored extension changes; every destination on a matching rule is delivered, and each delivery (success or failure) is recorded in the alert log.

Destinations come in several kinds:

Testing a destination sends a real notification. For a Jira or ServiceNow destination that means one real test ticket is created — proving the project/table, auth, and field mapping end-to-end.

Background refresh

Extensions on the watchlist are automatically re-fetched every ICEBERG_EBS_FETCH_INTERVAL_MINUTES minutes (default: 60). Re-scoring detects publisher changes, install count drops, and new code patterns. Toggle watchlist per extension on the detail page.

The dashboard Fetch health tile counts watchlist extensions that are failing (last fetch errored) or stale (no successful refresh recently), and the extension row flags each one. Transient store failures are retried with backoff before counting, and a store that fails repeatedly in a cycle is skipped as a store outage — which doesn't count against an extension, so a store being down doesn't spike the tile. Set ICEBERG_EBS_RETENTION_DAYS to prune fetch/install/alert history older than the window. Ops probes: /healthz (liveness) and /readyz (readiness — checks the database and reports last_scheduler_run so an external monitor can alert when the scheduler has stalled). Set ICEBERG_EBS_LOG_JSON=true for JSON logs.

Outbound proxy

All outbound traffic — store metadata fetching, package downloads, webhook alert delivery, and single sign-on (OIDC discovery, token, and userinfo requests to your IdP) — can be routed through a forward proxy. Admins configure routing live at Proxy in the rail (changes apply from the next outbound request, no restart); the ICEBERG_EBS_PROXY_* environment variables seed the initial values. Three modes: System (default — honour the standard HTTP_PROXY / HTTPS_PROXY / NO_PROXY environment variables), None (always connect directly), and Explicit (use the configured proxy URL unless the target matches the no-proxy list — hosts, domain suffixes, IPs, or CIDR ranges).

Proxy credentials are environment-only (ICEBERG_EBS_PROXY_USERNAME / ICEBERG_EBS_PROXY_PASSWORD): they are injected at send time and never stored in the database, shown in the UI, or written to logs. Webhook deliveries keep their SSRF defence — the connection goes to a pre-validated IP through the proxy (an IP-literal CONNECT), so a proxy that denies IP-literal tunnels needs an allowlist entry, and a no-proxy bypass for a webhook destination must use an IP or CIDR entry rather than its domain name. The connectivity test on the proxy page dials only the deployment's known egress targets (the extension stores, configured webhook origins, and each enabled IdP), never an arbitrary URL.
