# IcebergEBS - Paranoid about Chrome extensions

## Summary
Collect information about extensions for chromium type apps, Chrome, Edge, VSCode etc. and provide risk scoring. Signals considered: permissions, popularity/install count, publisher identity, staleness, code behaviour (eval/obfuscation), and external domains contacted.

## Environment, Frameworks and Libraries
App will always run on Python 3.14 or later.

### Python
- FastAPI
- SQLModel with `asyncpg` (PostgreSQL ‚Äî dev, test, and production; SQLite is not supported). `psycopg2-binary` (dev only) backs the Alembic CLI / sync-engine paths.
- HTTPX
- pytest + pytest-asyncio
- pydantic-settings (config from env vars)
- itsdangerous (session cookie signing)
- jinja2 + python-multipart (templates + form parsing)
- uvicorn[standard] (ASGI server)
- beautifulsoup4 (Chrome HTML scraping only)
- apscheduler (3.x stable ‚Äî 4.x is alpha-only, do not use)

### Dependency management (uv)
**Dependabot** (`.github/dependabot.yml`) opens weekly **grouped** update PRs across four ecosystems: `uv` (Python), `github-actions`, `docker` (Dockerfile base images), and `docker-compose` (the postgres/caddy pins). A Python PR updates `pyproject.toml` **and** `uv.lock` together, so `uv sync --locked` stays green. One trap worth knowing: Dependabot's Docker parser reads **`FROM` lines only** ‚Äî an image referenced via `COPY --from=<image>` is invisible to it, which is why the uv binary is pulled through a named `FROM ‚Ä¶ AS uv` stage in the Dockerfile. A second trap: Dependabot does **not** parse Helm `values.yaml` image tags, so the **Helm Caddy sidecar pin** (`helm/iceberg-ebs/values.yaml` `caddy.image.tag`) has no automated update path ‚Äî it's kept in lockstep with the Dependabot-managed **Compose** Caddy image by `tests/test_helm_caddy.py::test_helm_caddy_tag_matches_compose`, which fails when the two drift so a Compose bump forces the Helm bump in the same PR (#200). Bump both together.

`pyproject.toml` is the **only** dependency manifest ‚Äî there is no `requirements.txt`. Runtime packages go in `[project.dependencies]`; test + static-analysis tooling goes in the **`[dependency-groups] dev`** group (PEP 735). After changing either, run **`uv lock`** and commit the updated `uv.lock`: CI installs with `uv sync --locked`, which fails on a stale lock, and the `lint` job runs an explicit `uv lock --check`. IcebergEBS is a **virtual project** (`[tool.uv] package = false`) ‚Äî it is deployed as source + uvicorn, never built into a wheel, which is what lets the Dockerfile install the dependency layer from `pyproject.toml` + `uv.lock` alone. The production image builds its venv with `uv sync --frozen --no-dev`, so the `dev` group **cannot** reach the container: anything a runtime import needs must be a real runtime dependency, not a dev one.

## Architecture
API-first design. All data flows through FastAPI endpoints; the UI consumes them. HTML routes render Jinja2 templates; API routes return JSON.

### Key modules
- `app/main.py` ‚Äî FastAPI app + lifespan. Logging is configured by **`app/logging_config.py:setup_logging()`** (#89): timestamped text, or single-line JSON when `settings.log_json` (`ICEBERG_EBS_LOG_JSON`) is set ‚Äî replaces the old bare `basicConfig` (no timestamp). Unauthenticated ops probes (#26): **`/healthz`** (liveness ‚Äî process up, no dependency checks) and **`/readyz`** (readiness ‚Äî runs `SELECT 1`, returns 503 if the DB is unreachable; also reports `last_scheduler_run` from the in-process `app/scheduler_state.py` signal (no history scan on the probe; scheduler-only so an API fetch can't mask a stall) so an external monitor can catch a stalled scheduler ‚Äî advisory, never flips the 200/503 ‚Äî #89). Helm/Compose probes point at these. The dashboard (`routes/ui.py`) surfaces per-extension last fetch status/error and a fleet **"Fetch health"** count of watchlist extensions that are failing (latest `FetchLog.success` is false **and not** `store_outage`) or stale (no successful refresh within `max(2√ó fetch_interval, 60m)`) ‚Äî a store-outage skip (#108) is excluded from "failing" so a store being down doesn't blame its extensions (a prolonged outage still surfaces via staleness)
- `app/config.py` ‚Äî pydantic-settings `BaseSettings`, env prefix `ICEBERG_EBS_`
- `app/version.py` ‚Äî `get_version()` (lru_cached): resolves the running build as `build N ¬∑ sha` (`N` = `git rev-list --count --first-parent HEAD`, +1 per merge to main) in priority order `ICEBERG_EBS_VERSION` env ‚Üí stamped `app/_version` file ‚Üí runtime git ‚Üí `"dev"`. Shown at the bottom of the rail and set as the FastAPI app `version`. See the Versioning section below
- `app/database.py` ‚Äî async SQLAlchemy engine (PostgreSQL) with a tuned connection pool (`pool_size=5`, `max_overflow=10`, `pool_pre_ping`, `pool_recycle=1800`). **Schema is managed by Alembic** (D1): `init_db()` opens a connection and `run_sync`s `_run_migrations`, which upgrades an empty database to head from the baseline, or ‚Äî for a database created the old way (tables present, no `alembic_version`) ‚Äî **stamps** it at the *baseline* revision (without recreating anything) and then upgrades to head, so post-baseline migrations still run (#143; stamping head would have marked them applied without running them). A database already **falsely stamped** at head by the retired path (baseline schema behind a post-baseline stamp, detected via the still-naive `user.created_at` ‚Äî the first post-baseline migration made it timestamptz) is re-stamped at the baseline and repaired the same way. A **stamped-but-empty** database (`alembic_version` present but the core `user` table missing ‚Äî the state the test suite's `create_all`/`drop_all` used to leave a dev DB in, since `alembic_version` isn't a SQLModel table; #113) has its lying stamp reset to base so the upgrade rebuilds from scratch instead of trusting it and running nothing ‚Äî inert in production, which never drops tables. Conversely an **unstamped database with tables present** (no `alembic_version`) is classified by matching its live schema against the models: if it **exactly matches head** (`_schema_matches_head` via `compare_metadata`) ‚Äî what an *aborted* test run from the current checkout leaves behind, since conftest drops `alembic_version` in setup and `create_all` builds the head schema; #199 ‚Äî it is stamped at **head** so the upgrade is a no-op, instead of being misclassified as pre-Alembic, stamped at the baseline, and re-running post-baseline migrations against columns `create_all` already made (`DuplicateColumn`, crashing `make dev`). An unstamped schema that is **post-baseline but does *not* match head** (`_has_post_baseline_schema` true via the still-naive-`created_at` discriminator, but `compare_metadata` non-empty ‚Äî an aborted run from an *older* checkout, then the code updated) is **refused with a clear error** rather than stamped at head and silently skipping the intervening migrations (#199 review). Both are inert in production, which never runs `create_all` and whose genuine pre-Alembic DBs have the naive-`created_at` baseline schema. `tests/conftest.py`'s session fixture also drops `alembic_version` in setup + teardown so a `make test` no longer leaves the dev DB unbootable. The version INSERT is committed explicitly (`await conn.commit()`) because the connection isn't in autocommit, so the DML row would otherwise roll back when the connection closes. Migration env in `alembic/` (`env.py` reads `settings.database_url`, registers `SQLModel.metadata`, and reuses the startup connection via `config.attributes["connection"]`); baseline revision in `alembic/versions/`. **Adopting Alembic assumes existing production DBs are already on the final pre-Alembic schema** (they are ‚Äî the retired `_migrate_*` ran on every prior startup), which corresponds to the baseline revision, before being stamped. Add new schema changes with `alembic revision --autogenerate`; `tests/test_migrations.py::test_head_matches_models` fails if the baseline/head drifts from `models.py`.
- `app/models.py` ‚Äî SQLModel table definitions (`User`, `Extension`, `FetchLog`, `InstallCountHistory`, `InstallObservation`, `AlertDestination`, `AlertRule`, `AlertLog`, `ApiKey`). Timestamp columns use `sa_column=_tz_column(...)` (`timestamptz`) because the app writes tz-aware UTC datetimes. **Referential cleanup lives in the schema** via FK `ondelete` (the SQLModel-recommended pattern): history-preserving FKs are `SET NULL` (`AlertLog.{rule_id,destination_id,user_id}`, `Extension.user_id`), child rows removed with their parent are `CASCADE` (config rows + per-extension `FetchLog`/`InstallCountHistory`/`InstallObservation`/`AlertRule`/`AlertLog`). The delete handlers rely on these instead of hand-severing FKs ‚Äî see the FK migration `‚Ä¶_fk_ondelete.py`. **`InstallObservation`** (#29) is SOAR-fed org inventory: one row per `(extension_id, asset_id)` (unique constraint ‚Üí re-pushes upsert `last_seen`), carrying `asset_type`/`department`/`source`/`first_seen`. `Extension.install_footprint` caches the distinct-asset count, maintained by `POST /api/inventory`; **exposure** ("blast radius") = `risk_score √ó install_footprint` is always **derived** (never stored). **`FetchLog.store_outage`** (#108) marks a failure row written when the per-store circuit breaker skipped the extension (store down), so the Fetch-health tile can tell that apart from a broken extension. **`Extension` JSON-in-str columns** (`permissions`, `risk_detail`, `package_analysis`, `pending_alert_events`) are read through a single defensive parse each (missing / unparsable / wrong-shape ‚Üí safe fallback) via `utils.json_list`/`json_object`, so consumers never re-hand-roll it and can't reintroduce the #17/#61 bug class (#167). Three are typed accessor methods on `Extension` ‚Äî `permissions_list()`, `analysis_dict()`, `risk_detail_dict()`. The fourth, `pending_alert_events`, is decoded by **`services._parse_pending_events(raw)`** (not a model method ‚Äî it returns typed `ChangeEvent`s from `notifications.py`, which the model can't import without a cycle): it drops both non-dict entries **and** malformed event dicts, so a corrupt/partial marker can't crash the fetch or re-fire forever (#197). Writers still `json.dumps` back onto the column
- `app/auth.py` ‚Äî itsdangerous signed cookies, `require_auth` / `require_admin` FastAPI dependencies; `verify_credentials` always runs bcrypt (via `_DUMMY_HASH`) even for unknown usernames to prevent timing-based user enumeration. `hash_password` / `verify_password` are **async** ‚Äî bcrypt is ~100ms of pure CPU, so they offload to a worker thread via `anyio.to_thread.run_sync` (single uvicorn worker: running it inline stalls every request + the scheduler). All callers must `await` them; the sync cores are `_hash_password_sync` / `_verify_password_sync`. `require_api_auth` **throttles** the `ApiKey.last_used_at` write (only when missing or older than `settings.api_key_last_used_throttle_seconds`, default 60s) ‚Äî a commit per request (including read-only GETs) is a wasted write that contends with the scheduler under load. **Session revocation (M1):** every cookie carries a signed issued-at (read via `get_session_claims`); `require_auth`/`require_api_auth` reject cookies older than the user's `User.password_changed_at` (`_session_after_password_change`, 1s tolerance for timestamp granularity), and `change_password` bumps that marker **and** deletes the user's API keys ‚Äî so a password reset invalidates other-device sessions and leaked bearer tokens. **`require_admin_ui`** is the HTML-admin counterpart to `require_admin`: built on `require_auth` it gives 303 redirects (to `/login` if unauthenticated, `/` if non-admin) instead of raw JSON 401/403
- `app/fetchers/` ‚Äî one fetcher class per store; `get_fetcher(store, client)` factory in `__init__.py`. Package download is best-effort: `fetch()` (base + vscode/edge overrides) catches **only** `(FetchError, httpx.HTTPError)` ‚Üí logs + `package=None`; genuine programming errors propagate instead of being swallowed into a silent midpoint-fallback score (M5). **`transport.py` (#108)** holds `RetryTransport`, the shared client's resilience layer: it retries **transient** failures (connect/timeout + `RETRYABLE_STATUS` = 429/500/502/503/504) on **idempotent** methods only, with exponential backoff + full jitter, honouring `Retry-After` (bounded by `_MAX_SLEEP_SECONDS` so a hostile value can't wedge the sequential scheduler). 404 is never retried (delisted ‚Üí surface now); POST is never retried (webhook delivery must not be silently re-sent, and its call site already disables redirects). `transport.py` also holds **`ProxyRoutingTransport`** (#216): routes each request direct or through the configured outbound proxy by consulting `app/proxy.py`'s in-memory snapshot **per request** (so an admin edit applies on the next request ‚Äî no client rebuild), lazily building the proxied inner transport keyed on the credential-injected proxy URL and **retiring (not closing)** the old one on change (in-flight requests finish; everything closes in `aclose()`). It's a transport (not httpx `mounts=`) because NO_PROXY CIDR entries can't be URL patterns. Deliberately no logging inside it ‚Äî the resolved proxy URL carries credentials. `app/main.py` builds the chain `RetryTransport(ProxyRoutingTransport(limits=‚Ä¶))` ‚Äî retry OUTER so each attempt re-routes; limits go on the **innermost** transports because httpx ignores `AsyncClient(limits=‚Ä¶)` when a custom `transport=` is supplied (direct + proxied pools each carry `limits`, so the theoretical cap is 2√ó)
- `app/inspector.py` ‚Äî static analysis of downloaded zip packages (CRX/VSIX). `inspect_package()` is pure-CPU (~20 regexes over up to 500 JS files); `services.py` calls it via `anyio.to_thread.run_sync` so it doesn't stall the event loop / scheduler. `PackageAnalysis.to_json_dict()` (what `services.py` persists to `Extension.package_analysis`) and `PackageAnalysis.stored_defaults()` (what `routes/ui.py` backfills onto a sparse/old stored blob before rendering) are the **single home** for the stored-field list ‚Äî both derive from the dataclass fields (excluding the internal `_finding_keys` and the transient `version`/`author` manifest fallbacks), so a new analysis field is one dataclass edit and can't drift between store and render (#164); `tests/test_inspector.py` asserts the two key-sets stay identical
- `app/permissions.py` ‚Äî single source of truth for the permission-tier sets (`CRITICAL_PERMISSIONS`, `HIGH_PERMISSIONS`, `MEDIUM_PERMISSIONS`, `BROAD_HOST_PATTERNS`). Imported by both `scoring.py` (score) and `inspector.py` (findings) so the two can't drift (#63) ‚Äî add a permission to a tier here, never re-inline these sets
- `app/scoring.py` ‚Äî pure scoring functions, `compute_risk_score()` ‚Üí `RiskDetail` NamedTuple; `risk_level(score)` is the single source of truth for the score‚Üíband thresholds (75/50/25), reused by `routes/api.py` and `notifications.py` ‚Äî do not re-inline those thresholds elsewhere. Permission tiers come from `app/permissions.py`
- `app/services.py` ‚Äî `fetch_and_store(ext, session, client)`: shared pipeline used by both API routes and the scheduler; stages changes but does **not** commit and does **not** fire alerts ‚Äî it returns `(ext, events)`. `fire_pending_alerts(events, ext, engine, client)`: the caller invokes this **only after committing**. Firing is deferred on purpose: `fire_alerts` opens its own second DB session, which must not run inside the caller's still-open write transaction (it would contend with / be isolated from the caller's uncommitted writes). Never call `fire_alerts`/`fire_pending_alerts` while the caller's write transaction is open. After commit, callers `session.refresh(ext)` (reloading expired attrs) before firing, since `fire_alerts` reads `ext` to build the payload. **Recoverable alerts (#109):** `fetch_and_store` stages the detected events on `Extension.pending_alert_events` (JSON) in the **same commit** as the state change, **merging** with any still-undelivered events already in the marker rather than overwriting them (a prior failed delivery leaves them for retry, and the manual-refresh path doesn't run recovery first). It returns the full merged set so the caller fires everything pending. `fire_pending_alerts` clears the marker only after `fire_alerts` returns (delivered + logged), and only via **compare-and-clear** (`_clear_pending_alerts(ext_id, engine, expected)` clears iff the stored marker still equals `expected`) so a concurrent refresh's appended events aren't wiped. `expected` is the raw marker string the events were derived from: the normal path re-serialises the (unfiltered) fired events ‚Äî an identical string ‚Äî while recovery passes the **raw** marker explicitly (`fire_pending_alerts(..., expected_marker=raw)`), because it may have delivered a *filtered subset* of a corrupt marker and must still clear the whole thing, or it never matches and re-fires every cycle forever (#197). If the process dies between commit and delivery, the marker survives and **`recover_pending_alerts(engine, client)`** ‚Äî run at the head of each `refresh_watchlist` cycle, deliberately **not** in the lifespan at startup (running it inline there would block uvicorn from binding / answering probes for up to one webhook timeout per pending extension, risking a liveness-probe kill mid-recovery; deferring to the scheduler keeps startup fast **and** keeps recovery running in exactly one place so it can't race a concurrent refresh's delivery of the same events ‚Äî #155; the durable marker makes the ‚â§ `fetch_interval_minutes` deferral safe, nothing is lost) ‚Äî decodes it via `_parse_pending_events` (dropping any junk) and re-fires it (idempotent). **Graceful drain:** APScheduler 3.x's `AsyncIOExecutor.shutdown(wait=True)` *cancels* running async jobs rather than awaiting them, so the lifespan instead `scheduler.pause()`s, then `await drain_inflight(settings.shutdown_drain_seconds)` (which awaits the in-flight `refresh_watchlist` task tracked in `scheduler._inflight`), then `scheduler.shutdown(wait=False)`. The durable marker is the backstop past the container grace period (`terminationGracePeriodSeconds`/`stop_grace_period`, kept above `shutdown_drain_seconds`).
- `app/notifications.py` ‚Äî `detect_changes()` + `fire_alerts()`: compares old/new extension state and POSTs to matching webhook destinations (via `app.webhooks.send_webhook`); `fire_alerts()` takes an `AsyncEngine` and commits `AlertLog` rows in its own dedicated session, independent of the caller's transaction. It must run **after** the caller has committed (see `fire_pending_alerts`) so its session does not run inside the caller's open write transaction. It groups matching rules by event type and delivers **every** event against its rules ‚Äî including multiple events of the same type, which the merged pending marker can hold (e.g. `new_version` 1.0‚Üí1.1 then 1.1‚Üí1.2); it must **not** collapse to one event per type or the older undelivered event is dropped and then cleared with the marker (#144). The `permission_change` event diffs **both** API permissions (`ext.permissions`) **and** host permissions (read from the stored `package_analysis` via `_host_permissions`) ‚Äî gaining broad host access like `<all_urls>` must alert (#60); both fields are only rewritten on a fresh successful inspection, so a transient download failure can't fire a spurious change. **`build_alert_payload(...)`** is the single source of the on-the-wire webhook payload shape (`{text, event, extension:{id,name,store,store_url,[iceberg_ebs_url]}, change:{old,new}, risk_score}`), called by both `fire_alerts()` and the `alerts.test_destination` endpoint so the "test" webhook is the real webhook by construction and can't drift (#168)
- `app/webhooks.py` ‚Äî `validate_webhook_url()` (SSRF validation: scheme/blocklist/IP-range checks, returns the resolved public IPs) and `send_webhook()` (validates + resolves + connects to the pinned IP, preserving the original `Host` header and TLS SNI). DNS resolution is isolated in `_resolve_host()` so tests can stub it. Both `alerts.py` and `notifications.py` send through this module
- `app/scheduler.py` ‚Äî APScheduler `AsyncIOScheduler` background watchlist refresh; each extension is processed in its own `AsyncSession` + commit so a single failure cannot corrupt or roll back changes for the entire batch. `_refresh_one` returns an `_Outcome` (SUCCESS/FAILED/GONE) that drives a per-cycle, per-store **`_StoreCircuitBreaker`** (#108): a store's counter resets on any success, so an isolated broken extension (404) never trips it, but `settings.store_circuit_failure_threshold` **consecutive** failures open the store's circuit and the cycle's remaining same-store extensions are skipped and logged as `FetchLog.store_outage=True` (via `_record_store_outage`) rather than each looking broken. Also registers a **daily retention prune** (`run_retention_prune`) ‚Äî only when `settings.retention_days > 0`, firing once at startup (`next_run_time=now`, #145). Both jobs set `misfire_grace_time=None` so a busy/stalled single-worker loop can't silently drop a due fire under APScheduler's 1s default ‚Äî in particular the prune's at-startup fire on a CPU-starved restart (#198)
- `app/retention.py` ‚Äî data-retention pruning (#22). `prune_expired(session, retention_days, now=...)` deletes `FetchLog`/`InstallCountHistory`/`AlertLog` rows older than the window (cutoff = `now ‚àí retention_days`), returns per-table delete counts, and is a no-op when `retention_days <= 0`; `Extension` rows are never touched. `run_retention_prune()` is the scheduler entry point ‚Äî own session + commit, gated by `ICEBERG_EBS_RETENTION_DAYS` (default 0 = disabled)
- `app/proxy.py` / `app/proxy_settings.py` / `app/routes/proxy.py` ‚Äî **outbound proxy (#216)**. `proxy.py` is the pure resolver (typed, mypy-clean): `resolve_proxy_url(cfg, url) -> str | None` (None = direct) over three modes ‚Äî `SYSTEM` (default; parses `HTTP(S)_PROXY`/`ALL_PROXY`/`NO_PROXY` from the env itself, because httpx never applies env proxies through a custom `transport=`), `NONE`, `EXPLICIT` (configured URL unless the target matches the no-proxy list ‚Äî standard NO_PROXY semantics incl. CIDRs, `_should_bypass`). Routing is read from an in-memory `ProxyConfig` snapshot (`get_config`/`set_config`; `None` = unloaded ‚Üí direct, pre-feature behaviour) consulted per request by `ProxyRoutingTransport` (see `app/fetchers/` above), so **all** egress ‚Äî store fetchers, package downloads, webhook delivery ‚Äî inherits it. `proxy_settings.py` manages the admin-editable `ProxySettings` DB singleton (id=1), lazily seeded from the `ICEBERG_EBS_PROXY_*` env on first read; every save pushes a fresh snapshot. `routes/proxy.py` is the admin JSON API (GET/PUT settings ‚Äî PUT is `extra="forbid"` so credential keys 422, rejects a userinfo `proxy_url` (creds would persist onto the API-visible row), and enforces the EXPLICIT‚áíURL invariant against the **resulting** settings, not just the request fields ‚Äî clearing only `proxy_url` while the stored mode is explicit would otherwise silently fail open to direct; GET `/api/proxy/targets` returns **labels only**; POST `/api/proxy/test` dials a server-known target by label ‚Äî never a body URL, which would be an SSRF oracle ‚Äî with `follow_redirects=False` always, the full validate+pin delivery treatment for user-controlled webhook origins, and returns `ok: HTTP n` or `error: <ExceptionClassName>` only). **Credentials are env-only** (`proxy_username`/`proxy_password`): injected into the proxy URL at resolution time (`_with_credentials`), never on the DB row, never in API responses, scrubbed from log lines (`proxy.scrub`); `validate_proxy_settings` also rejects a userinfo env URL. Webhook interplay: `send_webhook` pins the URL netloc to the validated IP, so the routing decision (and any no-proxy bypass) runs against the **IP** ‚Äî bypassing a webhook destination needs an IP/CIDR entry, and the pinned CONNECT preserves the DNS-rebinding defence through the proxy. `validate_proxy_settings` runs first thing in the lifespan (fail-fast; error text never echoes the URL), and the startup `refresh_cache` is **fail-closed**: a load failure aborts startup rather than leaving the snapshot unloaded (unloaded = all-direct, which would let an EXPLICIT deployment silently bypass its mandated proxy). Admin page at `/admin/proxy` (changes apply on the next outbound request ‚Äî single worker, in-process snapshot)
- `app/ratelimit.py` ‚Äî `LoginRateLimiter` (process-local `login_limiter` singleton) for app-level login throttling (M3), independent of the edge proxy: counts failures per (client IP + username), locks the pair out for `settings.login_lockout_seconds` after `settings.login_max_attempts` within `login_attempt_window_seconds`. In-process state is fine because the deployment mandates a single worker. `login_post` returns 429 + `Retry-After` while locked and `reset()`s on success. Also `RequestRateLimiter` (token-bucket) backs **app-side edge rate limiting** ‚Äî the replacements for the old nginx `api` **and** `login` `limit_req` zones, added because stock Caddy has no `rate_limit` directive. The `edge_rate_limit` middleware in `app/main.py` applies two singletons keyed on `request.client.host` (the Caddy-set canonical `X-Forwarded-For`, spoof-proof per #77): `api_limiter` over `/api/*` (#188, gated by `settings.api_rate_limit_enabled`) and the tighter `login_request_limiter` over `POST /login` (#196, gated by `settings.login_rate_limit_enabled` ‚Äî its **own** switch, so disabling API limiting can't silently drop login protection; the failure-keyed `LoginRateLimiter` above only locks an (IP, username) pair after N failures and so stops neither username-spraying nor a bcrypt-cost flood). Both default **off** so the test suite isn't throttled; the Compose/Helm prod env enables them, with the cluster ingress limiting at the true edge too
- `app/deps.py` ‚Äî reusable FastAPI dependency aliases (`SessionDep`, `CurrentUser`, `WebUser`, `AdminUser`, `AdminUserUI`) used across route signatures instead of repeating `Annotated[..., Depends(...)]`. Each `/api` router declares its own `prefix="/api"` + `tags=[...]` on the `APIRouter` (so `main.py` just `include_router(...)` with no prefix, and `/docs` is grouped by tag). Endpoints declare their public schema as a **return-type annotation** (`-> ExtensionOut`, `-> list[UserOut]`, ‚Ä¶) rather than the `response_model=` decorator arg ‚Äî the FastAPI-preferred form, since the handlers already return the DTO. Also home to **`get_owned_or_404(session, model, obj_id, user_id, *, detail="Not found")`** (#169): the single owner-scoped fetch gate for the JSON API ‚Äî loads the row and 404s if it's missing **or** owned by someone else (indistinguishable, so ownership can't be probed), replacing the ~12√ó hand-inlined `session.get(...) + if not obj or obj.user_id != ‚Ä¶` across `routes/api.py`/`alerts.py`/`keys.py`. HTML routes that redirect-and-flash on a miss (`routes/ui.py`) keep their own handling

The filter+sort logic lives in **`app/extension_queries.py`** (see above); the list + export endpoints collect those params via the **`extension_filters` FastAPI dependency** (`FilterParams`) declared once (typed Query params ‚Üí 422 on bad input); the dashboard builds the same dataclass from its own coerced strings (tolerating junk). **`GET /api/extensions/export?format=csv|json` (#25):** streams the full (filtered, unpaginated) set of `EXPORT_FIELDS` (score + key fields, no nested findings/threat-intel) with a `Content-Disposition` attachment; shares the same filter/sort params. **Route order matters** ‚Äî `/extensions/export` and `/extensions/bulk` are registered before `/extensions/{ext_id}` so `"export"`/`"bulk"` aren't parsed as int ids. **`POST /extensions/bulk` (#24):** enrolls many extensions in one request (cap `MAX_BULK_ITEMS=100`), accepting structured `items` and/or a pasted `text` blob (`store,id` per line or store URLs via `_detect_store`/`_parse_bulk_text`). Each entry runs through the shared **`_enroll_extension`** primitive (validate ‚Üí dedupe ‚Üí create + `_fetch_and_score`, discarding the placeholder on a failed first fetch) ‚Äî the same helper `add_extension` now uses ‚Äî and returns a per-entry status (added/duplicate/invalid/error) + tallies. The loop captures `current_user.id` **before** processing because each enroll commits (expiring the ORM attribute). **`POST /api/inventory` (#29):** bulk-upserts SOAR install inventory (cap `MAX_INVENTORY_ITEMS=1000`); each observation resolves its extension through the same `_enroll_extension` primitive called with **`score=False`** (#78) ‚Äî so an **unknown** extension is auto-enrolled onto the watchlist but its **scoring is deferred to the scheduler** (which scores every `watchlist=True` extension on its next run), keeping a large batch of unknown extensions from doing hundreds of sequential store fetches inside one request. Newly-created rows report status `deferred`; already-tracked ones report `observed`. `_upsert_observation` then writes/refreshes an `InstallObservation` keyed on `(extension, asset)` via a Postgres `INSERT ‚Ä¶ ON CONFLICT DO UPDATE` (#76). After the batch, each touched extension's cached `install_footprint` (distinct asset count) is recomputed in one pass. **Exposure** is sortable everywhere via `sort=exposure` ‚Äî `SORT_COLUMNS["exposure"]` is the SQL expression `EXPOSURE_EXPR = Extension.risk_score * Extension.install_footprint` (NULL when either factor is, so existing nullslast handling applies; both in `app/extension_queries.py`); `ExtensionOut`/`EXPORT_FIELDS` expose `install_footprint` + `exposure` (computed by the `exposure()` helper mirroring the SQL)

- `app/alert_queries.py` ‚Äî the neutral home for **`get_alert_log(user_id, session, limit=50)`** (#149), the shared alert-log query + serialization consumed by both the JSON API (`routes/alerts.py`) and the dashboard render (`routes/ui.py`). - `app/routes/keys.py` ‚Äî JSON API routes for API-key management (`GET/POST /api/keys`, `DELETE /api/keys/{id}`). - `app/threat_intel.py` ‚Äî `build_threat_intel_indicators()`: turns the stored `package_analysis` (package hashes, external domains, network callouts) into indicator dicts with manual VirusTotal/OTX lookup links; output capped at `MAX_THREAT_INTEL_INDICATORS` (100, #28) without dropping the primary package indicators. No external TI APIs are called ‚Äî links only
- `app/middleware.py` ‚Äî `CSRFOriginMiddleware` (#107): Origin/Referer enforcement on state-changing browser requests (see Security below)
- `app/routes/users.py` ‚Äî JSON API routes for user management (`/api/users/...`). `delete_user` **preserves history** like `delete_rule`/`delete_destination` (#28): the user's config rows (rules, destinations, API keys) `CASCADE` away and the `AlertLog` FKs `SET NULL` via the schema's `ondelete` actions, so `AlertLog`/`FetchLog`/`InstallCountHistory` survive the account deletion. - `app/routes/ui.py` ‚Äî HTML routes, Jinja2 templates, flash messages. The **dashboard** does server-side filter/search/sort/pagination via the shared `build_extension_query` (page size 25), tolerating junk query params (falls back to defaults instead of 422); stat tiles use a lightweight column-only fleet snapshot, and a `qs(**overrides)` context helper builds filter/sort/page links that preserve state. It also renders a **"Top exposure"** section (#29) ‚Äî a column-only top-5 query ordered by `EXPOSURE_EXPR` over extensions with a known footprint ‚Äî and the **extension detail** page adds an **"Org footprint"** card (installed-on-N-assets + exposure + a per-department breakdown queried via `count(distinct asset_id) GROUP BY department`)
- `app/findings_view.py` ‚Äî `group_detection_findings()`: pure presentation logic that folds the flat stored `package_analysis["findings"]` into the grouped/sectioned shape the detail template renders (group per `(severity, source, code, title)`; rows deduped and shaped by whether a finding repeats across locations, details, or both). No HTTP/DB dependency ‚Äî extracted from `routes/ui.py` so it's unit-testable on its own (#170); `routes/ui.py` calls it when building the detail-page context

### Data flow
1. Caller (API route or scheduler) calls `fetch_and_store(ext, session, client)`
2. `fetch_and_store` calls `fetcher.fetch(extension_id)` ‚Üí `(ExtensionMetadata, bytes | None)`
3. If package bytes are present, `inspect_package()` runs static analysis
4. `compute_risk_score()` calculates the risk breakdown
5. Extension record, FetchLog, and InstallCountHistory are staged; `detect_changes()` compares the pre-fetch snapshot against the updated record; `fetch_and_store` returns `(ext, events)`
6. The caller **commits**, `session.refresh(ext)`, then calls `fire_pending_alerts(events, ext, engine, client)` ‚Üí `fire_alerts()` POSTs webhooks and commits `AlertLog` rows in its own session, decoupled from the caller's transaction. This ordering is mandatory: `fire_alerts`' second session must not run inside the caller's open write transaction

## Security
- Security of the application is a priority.
- Validate code to ensure there are no serious security flaws.
- Ensure authentication is applied to endpoints that shouldn't be public.
- `verify_credentials` in `auth.py` always pays the bcrypt cost via `_DUMMY_HASH` when the username doesn't exist ‚Äî never short-circuit before bcrypt runs or you leak username existence via timing.
- Webhook SSRF protection lives in `app/webhooks.py`. URLs are validated against a hostname blocklist (including subdomain suffixes) and IP range checks (`is_global`, `is_loopback`, `is_link_local`, `is_reserved`). Validation runs both at destination create/update time **and again at send time** in `send_webhook()`, which resolves the host and connects to the validated IP directly (IP pinning) ‚Äî closing the DNS-rebinding TOCTOU window between validation and the request. Redirects are disabled (`follow_redirects=False`) so a 3xx cannot bounce the POST to a private address.
- Jinja2 autoescaping is on by default ‚Äî do not disable it.
- Session cookies: HttpOnly + SameSite=Lax, signed with itsdangerous `URLSafeTimedSerializer`. Password change revokes other-device sessions + API keys (see `auth.py` note, M1). Login is throttled app-side by `app/ratelimit.py` (M3).
- **CSRF (deliberate, documented ‚Äî #16):** there are **no CSRF tokens**; protection is `SameSite=Lax` (+ `Secure` in prod) on the session cookie, and the JSON API requires an `application/json` body (which browsers can't send cross-origin via a plain form) with Bearer tokens as the primary M2M credential. This is a conscious trade-off, not an oversight (see the comment in `auth.py:set_session`). **Layered on top (#107):** `app/middleware.py`'s `CSRFOriginMiddleware` enforces an `Origin`/`Referer` check on **every** state-changing browser request (each non-safe method except Bearer-token M2M) ‚Äî including the unauthenticated `POST /login` that mints the cookie, so login CSRF is covered too (`origin_allowed()` matches the request host or `ICEBERG_EBS_TRUSTED_ORIGINS`). Bearer-token requests carry no browser `Origin` and are exempt. If even this needs strengthening, add per-request CSRF tokens in `set_session` + the templates rather than relying on SameSite alone.
- Never return raw exception text (`str(exc)`) to API callers on connection/SSRF paths ‚Äî it can leak resolved IPs / internal hostnames. Log the detail server-side and return a generic message (M4, `alerts.py:test_destination`, `routes/proxy.py:test_proxy`). `WebhookValidationError` messages are static and intentionally user-facing, so surfacing those at create/update time is fine.
- Outbound-proxy credentials (#216) are env-only secrets: never persist them to the DB, return them from an API, render them in a template, or log a resolved proxy URL (it carries them ‚Äî pass exception text through `proxy.scrub` first). The proxy connectivity test accepts only server-known target labels, never URLs (SSRF oracle), and webhook-origin targets dial the **origin only** (a Slack-style webhook path is a capability token that must not hit the wire or proxy logs on a test).

## Deployment

**Database:** **PostgreSQL only** ‚Äî dev, test, and production (SQLite support was removed). Postgres' row-level locking/MVCC lets the concurrent writers (scheduler + interactive API/UI + bulk ingestion) proceed without contention and scales the history tables. All writers are commit-isolated (scheduler, retention prune, and `fire_pending_alerts` after commit). Dev runs against a containerized Postgres (`docker-compose.dev.yml` / `make dev`); the Compose/Helm stacks provision it for production; rationale lives in `DEPLOYMENT.md ‚Üí Database choice`.

Full production deployment instructions are in `DEPLOYMENT.md`. Two options are covered:

**Docker Compose** ‚Äî three-service stack (postgres, app, caddy). **Caddy** (config in `caddy/`, #188 ‚Äî replaced nginx) terminates TLS and proxies everything ‚Äî including `/static`, served by the app's StaticFiles, since the built `output.css` exists only inside the app image (#85) ‚Äî to the app; it does **not** rate-limit (stock Caddy has no `rate_limit` directive), so edge throttling moved app-side (`app/ratelimit.py`, enabled via `ICEBERG_EBS_API_RATE_LIMIT_ENABLED` for `/api/*` and `ICEBERG_EBS_LOGIN_RATE_LIMIT_ENABLED` for `POST /login`, #196). Topology is **cluster nginx-ingress-controller (TLS via cert-manager, edge rate limiting) ‚Üí in-pod Caddy sidecar (:8080) ‚Üí app (localhost:8000)** (#188): the sidecar owns the canonical security headers via the `caddy` ConfigMap (a test-guarded mirror of `caddy/Caddyfile.k8s` + `caddy/headers.caddy` ‚Äî Helm can't read files above the chart), so the ingress no longer carries a duplicated CSP snippet. ## Versioning
IcebergEBS carries **two** identifiers, shown together at the bottom of the left rail as `v{semver} ¬∑ build N ¬∑ sha` (e.g. `v0.1.0b1 ¬∑ build 74 ¬∑ 8823e7a`). They answer different questions and must not be conflated:
- **SemVer** (`0.1.0b1`) is the **release** version ‚Äî the only thing that can express a breaking change, and what an API/SOAR consumer pins. Single source of truth: **`[project].version` in `pyproject.toml`**, read at runtime by `app/version.py:_semver()` via stdlib `tomllib` (never hardcoded, never duplicated). Git tags use the **SemVer spelling** of the same value (`0.1.0b1` ‚áÑ tag `v0.1.0-beta.1`) ‚Äî the PEP 440 / SemVer mapping and the release procedure live in **`docs/RELEASING.md`**; changes are recorded in **`CHANGELOG.md`** (Keep a Changelog).
- **`build N ¬∑ sha`** is the **build** identifier (`N` = `git rev-list --count --first-parent HEAD`, +1 per merge to main). It advances on every merge and is *not* a release.

Resolved by `app/version.py:get_version()` (cached once per process) in priority order: `ICEBERG_EBS_VERSION` env ‚Üí stamped `app/_version` file (git-ignored) ‚Üí runtime git + pyproject ‚Üí `"dev"`. The first two carry a complete string and so win **wholesale**. Injected into every page via `_render()` in `app/routes/ui.py`, rendered in the `rail_footer` block (`.rail-version` in `app.css`).
- **Bumping the version requires `uv lock`.** `uv.lock` records the project's *own* version, so a `pyproject.toml` bump without a lock refresh fails CI's `uv lock --check`. This is the most common way to break the build here.
- `_semver()` **never raises** ‚Äî a missing or malformed `pyproject.toml` degrades to the bare `build N ¬∑ sha`. It runs on every page render, so it must not be able to 500 the UI.
- **Auto-increment:** on the bare-uvicorn droplet (a git checkout) the number advances on each `git pull` of main ‚Äî no manual bump.
- **No `.git` (Docker/Helm):** `.dockerignore` strips `.git`, so the image relies on the `ICEBERG_EBS_VERSION` env (Dockerfile `ARG`/`ENV`). The `.github/workflows/build.yml` workflow computes the same string and passes it as `--build-arg ICEBERG_EBS_VERSION`. It checks out with `fetch-depth: 0` ‚Äî required, or `rev-list --count` is wrong on Actions' shallow clone.
- Keep the format string in sync between `app/version.py:_format()` and the version-compute step in **both** `build.yml` and `release.yml` ‚Äî all read the SemVer from `pyproject.toml`. A drift is invisible until a container deploy reports a different version from the droplet.
- **Releasing (#99):** pushing a `v*` SemVer tag triggers `release.yml`, which **verifies the tag matches `pyproject.toml`** (normalizing PEP 440 ‚áÑ SemVer, failing on mismatch), then builds a signed + attested release image (SBOM, SLSA provenance, cosign keyless) and cuts the GitHub Release. Release images ‚Äî not `build.yml`'s `:edge`/`:<sha>` dev images ‚Äî are the only deployable, verifiable artefacts; pin them by immutable tag/digest. Full procedure + `cosign verify` / `gh attestation verify` flow in `docs/RELEASING.md`.

## Contributing & the review bot
Merges to `main` go through the **`icebergai-review-bot`** (an automated reviewer) and `main` is protected ‚Äî you cannot push to it directly. Hard-won notes on working with it:

**How the bot behaves**
- Reviews the full `main...HEAD` diff, posts an **"IcebergAI Review Bot"** check-run plus a PR review with a verdict (`approve` / `request_changes`), and **merges the PR itself on approval**.
- It **auto-closes the linked issue only from a `Closes #N` keyword in the PR _body_.** A `(#N)` in the PR _title_ is a cross-reference, **not** a closing keyword ‚Äî the issue stays open. Always put `Closes #N` in the body, then after merge **reconcile**: confirm the issue actually closed, and if a title-only reference left it open, close it manually with a comment linking the merged PR.
- It re-reviews **every new head SHA** and compares files by **blob SHA across heads** ‚Äî a rebase that leaves a file's blob unchanged makes it re-raise the identical finding. Fix the code; a rebase alone won't clear a real finding.
- Its findings are usually real. On a `request_changes` P1, fix the root cause **and add the regression test it asks for** rather than pushing back ‚Äî it keeps blocking until the concern is genuinely addressed (e.g. #109 escalated through three distinct P1s: pending-alert overwrite ‚Üí fake shutdown drain ‚Üí non-atomic merge/clear race).
- **Runtime is flaky:** it can throw a RuntimeError ("‚Ä¶will be retried") and auto-retry, and a verdict can briefly oscillate or stall. Wait it out ‚Äî do **not** push to "un-stick" it.
- **"Dismiss stale reviews on push" is ON.** Any push ‚Äî including an empty "nudge" commit ‚Äî dismisses a fresh approval, and a nudge can race an incoming approval webhook and cancel it. **Never push to an approved (or freshly reviewed) PR unless you have a real change to make.**

**Branch & PR hygiene**
- One issue per branch, branched from **latest `main`** ‚Äî **never stack PRs.** GitHub's *documented* default is to **retarget** an open PR to its base's base branch when the base branch is deleted on merge ([docs](https://docs.github.com/en/pull-requests/collaborating-with-pull-requests/incorporating-changes-from-a-pull-request/merging-a-pull-request)), but stacking is fragile in practice: in this repo **#138 was auto-closed** (not cleanly retargeted) when its stacked base ‚Äî the #108 PR branch ‚Äî was deleted on merge, silently losing the work. Branch every PR from `main` so a base merge can't strand it.
- Clear `mergeable_state: behind` by rebasing onto `main`. Recurring conflicts land on the append-only shared files (`CHANGELOG.md`, `CLAUDE.md`, `DEPLOYMENT.md`) ‚Äî resolve by keeping **both** entries in order, not by dropping one side.
- `mergeable_state` cheatsheet: `clean` = ready to merge ¬∑ `blocked` = missing the required bot review ¬∑ `behind` = base moved, rebase ¬∑ `unstable` = a non-required check (often the bot review itself) still running.

**Running many PRs efficiently**
- CI reruns on every push and must be **green before the bot reviews**. Validate locally first when feasible (per the Testing section ‚Äî a real Postgres 18 + a venv installed from `pyproject.toml`) rather than spending review rounds on avoidable CI failures.
- Monitor with the PR-activity subscription / webhooks; pace status checks, don't tight-poll for events that arrive as notifications.
